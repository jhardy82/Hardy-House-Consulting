How should the presentation handle navigation?

Arrow keys + fullscreen work perfectly standalone — but the SPA's hash router and scroll system compete for the same key events when the deck is embedded.

A

Fullscreen-on-enter (Recommended)

User clicks → enters fullscreen mode
Section shows: [▶ Start Presentation] button
Inside fullscreen: arrow keys, F to exit → no conflicts

Clicking "Start Presentation" enters the Browser Fullscreen API. All key events are isolated. Matches how the source file already works. Press Escape or F exits. Section shows a preview/thumbnail before entry.

B

Click-to-advance (embedded, no fullscreen)

← Prev Slide 3 / 10 Next →
[Three.js shape animation]
Visible prev/next buttons · No keyboard conflicts

Render the deck inside the section with explicit prev/next buttons. Disable keyboard navigation to avoid conflicts. The presentation fills the section viewport but stays embedded. Simpler, but less immersive.

C

Focus-gated keyboard (embedded + keys when focused)

Click presentation to "focus" it
While focused: arrow keys advance slides
Click outside → keys return to SPA scroll

A tabindex container captures key events only when the user has clicked into the presentation area. Focus ring shows active state. Escape releases focus. Most ergonomic if you want to stay embedded with keyboard nav.
