01

Current state: seven isolated files

Every file is self-contained. Each works. None knows about the others except through the hub launcher. There is no shared navigation state, no persistent user journey, no progression between discovery and deep-dive, and no "story" that builds across files.

The core problem

A first-time visitor has no path. They land on the hub, see 7 cards, and must decide where to go with no context. There is no answer to "why does sacred geometry matter to a consulting firm?" before they are asked to explore it.

02

Proposed information architecture

A single HTML shell with hash-based routing. Every existing file becomes a loadable module within the shell. The shell owns navigation, persistent state, and the onboarding path. No file:// origin issues. No server required.

"The structure of the app should mirror the structure of the geometry itself — one circle at the centre, everything else radiating outward."

Navigation model

Single sticky nav bar. Hash routing (no server). Clicking a nav item loads that section's canvas(es), fires a CSS transition, and updates the URL. Back button works. Bookmark-able. The shell remembers which section you were in via localStorage.

State model

Three persistent pieces: current section, element affinity (set by the Oracle quiz), and journey progress (which steps completed). Element affinity changes the accent colour and default shape throughout the entire app. If your element is Water, the Icosahedron is pre-selected everywhere.

04

Entertaining introduction mechanics

Sacred geometry is not explained — it is experienced. Each mechanic is designed so the visitor discovers a truth by doing something, not by reading about it. The insight lands before the label does.

Mechanic A — The Breathing Circle

Land on nothing

The page opens to pure void. A single point pulses at the cursor. The visitor clicks. A circle expands, perfectly smooth, gold on black. Text appears: "This is all geometry needs to begin." Then a second circle appears. They overlap. The vesica piscis glows. No instructions. Just discovery.

Interaction: click to expand Duration: 45 seconds

Mechanic B — Element Oracle

Five questions, one solid

Five binary questions about how you work: "Do you initiate or complete?" / "Do you need structure or do you create it?" / "Does your best work happen in motion or in stillness?" Each answer spins a roulette of shapes. The fifth answer locks one solid as your element. It then rotates on screen, and all subsequent UI accents shift to its colour.

Interaction: choice-based Output: personalised element

Mechanic C — The Phi Hunt

Find φ in the wild

A gallery of six images — a nautilus, a sunflower, a galaxy arm, a hurricane, a pine cone, a human face — overlaid with golden ratio guides that the visitor reveals by hovering. The ratio counter ticks up each time they find one. At six: the golden spiral unfurls in 3D and the connection to r(θ) = base · φ^(2θ/π) is shown.

Interaction: hover reveal Count: 6 discoveries

Mechanic D — Build the Solid

Construct from scratch

Visitor is given a set of triangular face tiles and asked to assemble a tetrahedron by dragging them into position. The faces snap and lock when placed correctly. When the fourth face closes, the solid spins up in 3D and the element label appears. Five progressively harder builds: Tetra → Cube → Octa → Icosa → Dodeca.

Interaction: drag-and-drop Complexity: progressive

Mechanic E — The Edge Reveal

Hidden inside Metatron

Full Metatron's Cube shown dim. Visitor clicks a solid name from a list. The exact subset of the 78 edges that form that solid highlights progressively — each edge drawing itself in ~35ms intervals, just like the decomposition demo. When complete, the 3D solid materialises beside the flat pattern. The connection is visceral: you see the solid emerge from the lines.

Uses: decomposition.html logic Narration: per-edge

Mechanic F — Fibonacci Counter

Grow the sequence live

An interactive phyllotaxis spiral. A slider controls N (number of seeds). At N=3 it's a triangle. N=5 a pentagon. N=8 a cube ring. N=13 an icosahedron. N=21 a dodecahedron. The seeds spin and reorganise in real time. At each Fibonacci number, a petal sound plays and the corresponding team shape lights up — live proof that Fibonacci sizing is geometric.

Interaction: slider Maps to: GCMT team sizes

Sacred tree workflow patterns

The Tree of Life (Kabbalah, 10 Sephiroth, 22 paths, 3 pillars) maps exactly to ContextForge's dimensional framework. It is the oldest known model of emergent hierarchical process — and it predicts multi-agent coordination topology with remarkable precision.

"Every workflow is a tree. Every tree has geometry. Sacred geometry reveals the structure of the tree before you build it."

Tree 1 — Tree of Life (Kabbalistic)

10 nodes, 22 paths, 3 pillars

  • Kether (Crown) → Apex Agent / Strategic vision
  • Chokmah + Binah → Left/Right orchestration pair
  • Chesed + Geburah → Expansion vs Constraint agents
  • Tiphareth (Heart) → Integration / synthesis agent
  • Netzach + Hod → Creative and analytical arms
  • Yesod → Memory / context store (UCL)
  • Malkuth (Kingdom) → Execution / delivery layer

3 pillars map directly: Pillar of Severity (left) = Constraint agents. Pillar of Mercy (right) = Expansion agents. Middle Pillar = The coordinating chain from vision to delivery.

Tree 2 — Fractal Workflow Tree

Self-similar at every scale

Every task contains a micro-PAOAL. Every sprint contains a PAOAL. Every engagement contains a PAOAL. The workflow tree is fractal — the same pattern at every zoom level.

  • Trunk → Strategic engagement objective
  • Primary branches → Epics / work streams
  • Secondary branches → Sprints / cycles
  • Leaves → Individual tasks
  • Root system → Context and constraints (COF)

The L-system builder lets you set branching angle, trunk length, recursion depth, and Golden Ratio mode. In GR mode, every branch is φ times the length of the one that spawned it.

Tree 3 — ContextForge Agent Graph

The Pentagon as tree root

Visualise a live ContextForge workflow: Pentagon agents as nodes in a force-directed graph. Edges are baton handoffs. Thickness encodes handoff frequency. Colour encodes agent element. The graph responds to simulated workflow events in real time.

  • Nodes: apex + 5 Pentagon agents
  • Edges: 22 handoff types (mirroring Sephiroth paths)
  • Animate: baton pulse along active edge
  • Tooltip: UCL context snapshot per node

Tree 4 — PAOAL on Torus

The cycle as geometry

The PAOAL cycle (Plan → Act → Observe → Assess → Learn) mapped onto a torus surface. A particle traces the surface, completing one cycle per revolution. Inner loop = fast iteration. Outer loop = strategic arc. The hole in the centre is the Void — pure potential.

  • Interactive: drag to slow/speed
  • Annotate: each phase lights a zone
  • Counter: cycle count displayed
  • Overlay: current phase label

Tree 5 — Sri Yantra workflow

9 triangles, 43 sub-triangles

The Sri Yantra encodes 9 interlocking triangles (4 pointing up = Shiva/masculine, 5 pointing down = Shakti/feminine) producing 43 sub-triangles and 4 outer rings. Map this to: 9 = three PAOAL-phase clusters. 4 outer rings = 4 project phases. 43 sub-triangles = decision points in a complex engagement.

  • Click a triangle → reveals sub-workflow
  • Animate construction from inside out
  • ContextForge mapping overlay toggle

07

Key design decisions

Single file vs SPA

Single self-contained HTML with hash routing. No build step. No Node server. Open from file system or any static host. The existing pattern that already works.

Element as identity

The Oracle assigns one of 5 elements. This sets a CSS accent colour, a default shape, and a persistent label. It makes the tool feel personal across every section without requiring a login.

Journey is optional

New visitors get the 6-step path. Returning visitors (detected via localStorage) land directly on their last section. The nav is always visible — no lock-in to the journey.

One Three.js load

Currently each file loads Three.js independently. In the unified app, it loads once. All sections share the same renderer pool. Performance improves dramatically at scale.
