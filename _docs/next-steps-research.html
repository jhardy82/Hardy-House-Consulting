App Shell — Hash routing, shared state, single entry point

Pattern: single HTML + hashchange event

No framework or build step needed. The shell loads once; each section is a <div id="section-name"> toggled via display. Three.js loads once globally. All existing HTML files become inlined sections or dynamically-fetched partials. The hashchange event handles navigation; localStorage persists element affinity and journey progress.

// Core routing pattern
window.addEventListener('hashchange', () => {
  const id = location.hash.slice(1) || 'home';
  document.querySelectorAll('.section').forEach(s => s.hidden = s.id !== id);
  initSection(id);   // lazy-init Three.js scenes only when needed
});
// Persist element assignment

// CSS variable switching for element accent colour

Pattern: weighted scoring across 5 binary axes

Each question targets one axis: Initiate vs Complete, Structure vs Improvise, Motion vs Stillness, Detail vs Vision, Alone vs Collective. Each answer increments a score for one element. Phyllotaxis Slider — Fibonacci spirals and golden angle

Formula: θ = n × 137.5°, r = c√n

The golden angle (137.5076…° = 360°/φ²) is the only divergence angle that produces a fully packed non-overlapping spiral. At exactly 137.5° the result looks like a sunflower. A deviation of 0.1° in either direction produces visible straight-arm spokes instead — this contrast is the "wow moment". The slider runs from 1° to 180°. Fibonacci numbers on the scale (2, 3, 5, 8, 13, 21, 34, 55, 89) get tick marks mapped to GCMT team sizes.

function drawPhyllotaxis(angle_deg, N=500) {
  const ga = angle_deg * Math.PI / 180;
  ctx.clearRect(0,0,W,H);
  for (let n = 0; n < N; n++) {
    const theta = n * ga;
    const r = SCALE * Math.sqrt(n);
    const x = cx + r * Math.cos(theta);
    const y = cy + r * Math.sin(theta);
    ctx.beginPath();
    ctx.arc(x, y, DOT_R, 0, Math.PI*2);
    ctx.fillStyle = seedColor(n);  // colour by Fibonacci ring
    ctx.fill();
  }
}
Key insight
At 137.5°: sunflower. At 137.6°: faint spokes appear. At 138°: clear 5-arm star. This is the most dramatic single-slider demo possible.
GCMT mapping
Fibonacci numbers on the N axis light up the team shape: 5 seeds = Pentagon, 8 = Cube ring, 13 = Icosahedron, 21 = Dodecahedron.
Effort: 0.5 session · Canvas 2D only

L-System Fractal Tree — Lindenmayer system with turtle graphics

Stack-based turtle: [ push, ] pop, +/- rotate, F draw

The L-system is a string-rewriting formalism. Starting axiom: F. Production rule: F → F[+F]F[-F]F. After 5 iterations, the string encodes thousands of branch instructions. A turtle graphics interpreter walks the string, drawing lines and rotating. Branching is implemented via ctx.save()/ctx.restore(). The golden angle (137.5°) as the branching angle produces the most natural-looking trees. The φ-mode toggle sets the child branch length to parent/φ at each level — produces true Fibonacci proportions.

function drawTree(ctx, length, angle, depth) {
  if (depth === 0) return;
  ctx.beginPath();
  ctx.moveTo(0, 0);
  ctx.lineTo(0, -length);
  ctx.stroke();
  ctx.translate(0, -length);
  // Right branch
  ctx.save(); ctx.rotate(angle);
  drawTree(ctx, length / PHI, angle, depth - 1);
  ctx.restore();
  // Left branch
  ctx.save(); ctx.rotate(-angle);
  drawTree(ctx, length / PHI, angle, depth - 1);
  ctx.restore();
}
Sliders
Branching angle (15°–90°), depth (2–9), length ratio (0.5–0.8), φ-mode toggle, randomness ±N°
Golden angle mode
Set angle to 137.5° → phyllotaxis on a tree. The branches fan into sunflower-like spirals instead of a symmetric tree.
ContextForge mapping
Depth = hierarchy level. Trunk = strategic objective. Branches = epics. Leaves = tasks. φ-mode = all sub-work is scaled by the golden ratio from its parent.
Effort: 0.5–1 session · Canvas 2D only

ContextForge Agent Graph — Force-directed live workflow simulation

Option A: custom spring solver (lightweight) · Option B: vasturiano's 3d-force-graph (CDN)

Option A (recommended): A minimal custom Verlet spring solver runs in ~50 lines. 6 nodes (apex + 5 Pentagon agents), 10–15 edges. Each tick: apply repulsion (inverse square), apply spring forces along edges, dampen velocity, update positions. Renders as Three.js spheres + lines. No extra library. At equilibrium the graph self-organises toward the Dodecahedron vertex projection — this is the demonstration.

Option B: vasturiano/3d-force-graph (CDN: unpkg.com) wraps Three.js + d3-force-3d. Accepts JSON {nodes, links}. Supports DAG mode (top-down hierarchy). Directional arrows on edges. Click callbacks. This is production-grade with 5k+ stars and active maintenance. The cost is ~250KB extra from CDN.

// Custom spring solver — core loop (~50 lines total)
nodes.forEach(a => {
  nodes.forEach(b => {
    if (a===b) return;
    const dx=b.x-a.x, dy=b.y-a.y, dz=b.z-a.z;
    const d = Math.sqrt(dx*dx+dy*dy+dz*dz) || 0.001;
    const f = REPULSE / (d*d);
    a.vx -= f*dx/d;  a.vy -= f*dy/d;  a.vz -= f*dz/d;
  });
});
edges.forEach(({a,b,rest}) => {
  const dx=b.x-a.x, dy=b.y-a.y, dz=b.z-a.z;
  const d = Math.sqrt(dx*dx+dy*dy+dz*dz);
  const f = SPRING * (d - rest);
  a.vx += f*dx/d;  // pull toward rest length
  b.vx -= f*dx/d;
});
Baton pulse animation
When a handoff occurs, a particle travels along the edge from source to target over 1.5s, coloured by the handoff type.
Self-organisation result
With 5 Pentagon agents at equal strength and one apex node, equilibrium produces a pentagonal ring with the apex centred above — the Dodecahedron's cap projection. No hard-coding required.

Sri Yantra — geometry research

The Sri Yantra is mathematically the most complex pattern in the set. Research confirms it is an under-determined system — there are infinitely many valid configurations. The "Classical" Sri Yantra is defined by a specific set of verified real-number coordinates. The only correct implementation path is pre-computed coordinates, not runtime construction.

Why it's hard

Under-determined Euclidean problem

Nine interlocking triangles (4 upward, 5 downward) must satisfy three simultaneous constraints: Concurrency (all triple-intersection points must be exact points, not polygons), Concentricity (inner circle concentric with outer), and an equilateral inner triangle. Previous computer implementations all failed the concurrency constraint, producing small polygons at intersections instead of points.

The correct solution (Gérard Huet, Theoretical Computer Science 281, 2002) encodes the Sri Yantra as a system of line-line intersection equations and solves for the 9 key y-coordinates numerically.

Effort: 1–2 sessions · Pre-computed coords required

Implementation path

Use the TeXample verified coordinate set

A verified TikZ implementation (texample.net, 2025) provides pixel-accurate coordinates derived from Huet's equations, with correct superposition at all intersection points. The approach:

  • Translate the verified coordinate set to Canvas 2D (scale + y-invert)
  • Draw 9 triangles sequentially with staggered animation (35ms per triangle, matching the Metatron edge-reveal pattern)
  • Outer rings: 3 concentric circles + 2 lotus petal rings (8 and 16 petals)
  • Square outer frame (Bhupura) with 4 gates
  • ContextForge overlay toggle: colour each triangle by its PAOAL phase
Key coord
Center at (0,0), outer circle r=1. Inner triangle vertices at known offsets from Huet's solution.
Animation
Construct from innermost triangle outward — the reverse of how it's traditionally drawn but more visually dramatic.

Mechanic D — Build the Solid: revised scope

Simplification: assemble edges, not face tiles

Dragging face tiles in 3D with snap detection is genuinely hard (2 sessions). A simpler and equally satisfying version: display the solid's vertices as glowing spheres and give the visitor a set of unconnected edge "sticks" to drag between the correct vertex pairs. When the correct edge snaps into place, it locks gold and the count updates. Completing all edges triggers the face-fill animation. This reduces to point-to-point distance checks rather than polygon-in-3D-space detection — tractable in 1 session.

04

Risks and unknowns

Risk 1 — Sri Yantra coordinate accuracy

Medium risk

The TeXample coordinates are from a TikZ implementation, not a Canvas 2D one. Translating between coordinate systems (scale, y-invert, origin shift) must be done precisely or the concurrency constraint fails (intersections become polygons). Mitigation: validate each triangle-pair intersection visually before moving on. Expected: 1 re-iteration.

Risk 2 — Agent graph self-organisation

Low risk

The claim that a 6-node force graph with Pentagon symmetry self-organises to the Dodecahedron cap projection assumes equal repulsion and equal spring constants. In practice, slight asymmetries in initial positions or floating-point accumulation may produce a rotated or distorted Pentagon. Mitigation: apply angular constraints (each Pentagon agent constrained to 72° increments) in addition to spring forces.

Risk 3 — Single-file app size

Low risk

Inlining all 7 existing HTML files plus all new sections into a single file produces a 400–600KB HTML file. This is fine for a local file but may feel slow on first cloud/CDN load. Mitigation: hash-based lazy-loading (fetch section HTML on first visit to that section, cache in memory). All Three.js scenes remain lazy-init regardless.
